# PR Checklist v26 (Updated 19/06/26)

- Each checklist item below should be evaluated and checked or unchecked by reviewers, not by the pull request author. The only checklist item that needs to be evaluated by the pull request author is the "PR Title, Merges, Components, outdated CI and SemVer".
- Each checklist item should be addressed with enough detail so that the reviewer can quickly determine whether the item passes or fails. This information is especially helpful for reviewers who are not following the ticket closely.
- The checkbox before each checklist item is meant for the reviewers to evaluate the check, it does NOT imply that the checklist item must be addressed with a Yes or No answer.

| Code | Name | Description |
| ---- | ---- | ----------- |
| CPS001 | Backwards-compatible data access | Your changes are compatible with existing data in production OR appropriate migration logic has been included. Dropping of existing columns is known to break previous versions of code still running side-by-side during deployment. <br/><br/> When existing column have unique index added please make sure there is no duplicates in Staging and Production before completing. <br/><br/> Domain Events must be considered as persistent data as well and extra care about backwards compatible deserialization is needed when changing them, especially when Event Upgraders are involved. <br/><br/> Make sure that code changes, including package updates, do not break any existing Hangfire background jobs in the Hangfire storage. For example, the method that will be called when the enqueued job executes no longer exist or its signature has been changed, which will make the background job fail. This check also applies to background jobs created indirectly, for example, Hangfire jobs created using EventFlow.HangFire package. <br/><br/> Please consider points from CPS004 as part of this item. |
| CPS002 | Safe Multi-tenant EF migrations | Any EF migrations will not conflict with tenant isolation (aka Row Level Security), e.g. referential constrains that need to do a pre-check or unique indexes that may find non-unique existing data. Another potential problem is creating unique index on a column which will create a unique value across tenants - it should be combined key with TenantId instead. This does not apply to newly created tables but only to tables that will have existing data when deployed. <br/><br/> We should also pay attention to some hard to predict performance issues related to EF migrations. E.g. when adding a column to an existing table the command may not finish within allocated 30s timeout depending on live data, indexes, concurrent transactions and existing row locks. Therefore, it is recommended to separate out the schema changes into a technical PR and deploy it to production ahead of time. EF migrations that add new columns only should include the migration file and .Designer.cs file, do not include the model snapshot changes. <br/><br/> When the database migration script is executed manually, typically by the DevOps team, make sure that any new tables created are owned by the correct user and any RLS policies are created properly. See [this comment](https://EShopinsure.atlassian.net/browse/EShop-59087?focusedCommentId=311196) for more details. |
| CPS003 | API backwards-compatible for Clients | This PR can run without errors together with Backoffice Portal from master branch, i.e. no public API changes, no json:api resource changes, no new required fields. Please also note other clients like external applications, e.g. direct sales websites, aggregators etc. Please add an explanation of how clients may be affected in PR description. <br/><br/> Integration Events are part of public contract between a microservice and other components/clients. <br/><br/> Both API and messages need to be considered not only for attribute changes but also for cases when existing fields are handled differently, for instance previously valid values are now considered as error. |
| CPS004 | Compatible with service rolling updates | Because of rolling update strategy multiple instances of the service will be present in an environment when scaled out. Old and new version of modified microservice should work when deployed side by side to such environment - both processing requests and accessing the same (possibly migrated) database. <br/><br/> All domain Event Upgraders should be considered a BREAKING change because old version of the service will crash if it tries to read event created by the new version. |
| CPS005 | Independent deployment of each service | Each microservice modified in the PR can be deployed on its own and will work with older versions of other services in case they are not deployed at the same time. Please consider if it wouldn’t be better to split work into PRs with only service modified in each, especially when there are dependencies appearing. Otherwise what works locally is likely to break when only one service out of all changed is deployed. Please tick if only one service included. <br/><br/> Old version of each service must be working correctly with new version of other services and vice versa if the pull request deploys multiple services. |
| CPS006 | Safe release of new Features | What if the pull request creates critical problems in the production environment, e.g. crashing services, creating bad data, etc.? Please consider following options so that any problems like that can be resolved quickly and effectively if they ever happen. <br/><br/> - Adding a Feature Switch, disabled by default, for changes that are bringing on a new functionality for selected tenants. Disabled by default Feature Switches are also a preferred way of completing changes that are part of a bigger feature which cannot be delivered in one go. Functionality that requires significant changes across more than one microservice is very likely to need a switch. <br/> - Please note that feature switches is not the only mechanism for ensuring new code is released safely. Other common scenarios include purely additive functions that are not automatically active until explicit action from user to start interacting with them (e.g. new “button”, actions protected with new permissions or new Product Builder features). <br/> - For some other cases, especially background processes, we should also consider an Emergency Off Switch which would allow us to stop new code without any deployment (e.g. optional IConfiguration flag). <br/> - Finally, for most other cases we should have fallback with graceful feature degradation instead of errors. |
| CPS007 | Unit tests coverage | All new functionality should be coverred by BDD tests and/or low level unit tests. Please consider if any existing tests should be updated but preferably functionality should be structured according to business requirements even if it means running similar steps and/or code. It is very important to note that tests are not only a quality gate, regression prevention but also form a living documentation for current behavior of the system. <br/><br/> Please note that CI builds reports code coverage which should be maintained on high level which is enforced by CI builds. All added code should be checked for coverage in this report. <br/><br/> Please explain why no tests are needed - this should be an exceptional case. <br/><br/> Please consider this point together with CPS008. |
| CPS008 | Integration tests | Please consider if your changes require new or updated code in Integration Tests. Integration Test (IT) should not try to cover all scenarios and instead it should priorities: functionality that needs to access real database, confirming interactions between microservices and critical functionality. <br/><br/> Please tick if considered but decided no changes to integration tests are needed. |
| CPS009 | Exceptions and validations | This checklist item is not intended to be answered by listing exceptions or validations that were added or modified in the pull request. It is about whether exceptions are translated to the correct HTTP response status code for the client, including exceptions thrown in action methods and anywhere else in the HTTP request processing pipeline. <br/><br/> When throwing exceptions you should consider whether the result is a system error (HTTP 500) or a user error (HTTP 400 or 422, i.e. "validation" as defined in EShop.Shared.JsonApi.ExtendedErrorHandler). For example, an invalid request that fails validations should use `ArgumentException` or one of its derived types so that it translates to HTTP 400 or 422. `InvalidOperationException` should not be used in that case because it translates to HTTP 500 (Internal Server Error). <br/><br/> Please pay extra attention to external services because they may often throw a generic exception while the information about the error can be inside the actual response or specialized exception class. |
| CPS010 | Logging | This checklist item is not intended to be answered by pointing to new or modified logging source code. It is about whether logging has been considered and updated appropriately for relevant new or modified source code. Avoid double logging with Log + Throw because unhandled exceptions are already logged by default and Log + Throw will create unexpected duplicate log messages. <br/><br/> Please use Warning level for potential issue, Information for happy path and general overview, Debug for more details and Trace for a replacement of attached debugger (Trace is usually switched off in environments and Debug may be disabled) |
| CPS011 | Code analysis warnings | Please monitor compiler and refactoring warnings (and optionally refactoring suggestions as well) considering that we aim to enable Warnings as Errors in the near future and in the meantime keep the number of warnings at zero. Please note that Visual Studio shows warnings and suggestions (Information) only for open files. You can either check output of CI build for complete list of warnings, results from GitHub Code Quality analysis or execute full analysis on demand locally in Visual Studio (right-click solution →  Analyze and Code Cleanup → Run Code Analysis on Solution). <br/><br/> CI build analysis from GitHub Code Quality takes into account changes and highlights issues compared to version in master branch. While there might be some technical debt with existing issues we do not want to introduce any new items going forward. Please see more details in a section below. |
| CPS012 | Documentation (including feature recordings) | Code changes may require some form of documentation. Please make sure any relevant documentation has been updated (e.g. manual steps for developers, samples on how to use new features or simply notifying team members). <br/><br/> We should consider the videos of new feature as part of standard documentation, with goal to have one per AC as long as it is non-technical and understandable by business users. These should be in the relevant BDD feature files as well as in PR description. We don't need to store every recording that is added as AC verification. Instead we want those that provide good overview of business features. Bug fixes are unlikely to be good candidates for long term documentation like this. <br/><br/> Please consider optimizing video for size, e.g. Best Sampling, reduced colors, global color table, no transparency, detecting unchanged pixels and lowering frame rate where possible. <br/><br/> Ensure documentation files (.md, etc.) are organized according to the guidelines in Docs/ReadMe.md. |
| CPS013 | SemVer | Semantic version of all modified microservices has been updated (including changes in shared libraries; pure refactoring without any change in system behavior can rely on automatic build number). |
| CPS014 | PR Title | PR title is clear and readable for deployment team and contains all necessary codes <br/><br/> - [EShop-XXXXX] (for one or more affected tickets). <br/> - [HOTFIX] (for urgent bug fix that needs to be prioritized in the deployment). <br/> - [BUG] for bugs that do not need to get priority treatment. <br/> - [BREAKING] for any changes that require manual synchronization during deployment. <br/> - [MANUAL ACTION] for any actions that need to be done before or during or after deployment, e.g. database migration scripts that need to be run first, manual configuration updates, post-deployment checks, etc. When this tag is used, relevant sub-tasks with sufficient description of what need to be done should be created in the corresponding JIRA ticket and assigned to people who will execute them. <br/> - [Technical] for refactoring that does not change any functionality. <br/> - [Regression] for non-functional changes that may affect existing features. <br/> - [SWITCH] for releases that include a Feature Switch, which should be disabled by default. <br/> - [DEV TEST ONLY] for releases that do not require QC testing. <br/> - Similar mechanism exists in new Product generator changes which often affect many services through Shared code but are usually safe to deploy when functionality is fully additive, i.e. new code only kicks in when new features are configured in Product Builder - please use [PRODUCT BUILDER] tag in title for those cases. <br/><br/> **Please note that the title will be read by people outside of our dev team and might form part of release notes.** With that in mind, please describe in a brief and non-technical terms what is being released and not simply repeat the ticket title. <br/><br/> If PR includes more than one ticket please add all relevant SAAS codes. If the PR is related to a partial technical ticket please add both SAAS code for this ticket and for the parent. <br/><br/> If a hotfix is implemented under a new ticket (not the same number as one it is fixing) then please specify the original ticket as well with format “[HOTFIX for X]“, e.g. <br/><br/> PR 5761: “[HOTFIX for EShop-4148] [EShop-15877] Update tenant basic data to Json API controller” <br/><br/> The goal is to make it clear that this change fixes previous release labeled <br/><br/> PR 5648: “[Technical] [EShop-4148] JADNC upgrade in Tenancy” <br/><br/> Another case for hotfix could be for code that has already reached production - please use [LIVE HOTFIX] |

| CPS019 | Shared Product Generator | When new features are added to Shared, especially in Product Generator area, we have to make sure that all relevant services are updated. None of these dependencies are detected by our tools so usually it requires developer to analyze impact and update SemVer for affected services even if there are no other changes for them. <br/><br/> For Product Generator related changes it is almost certain that we should be updating: Configuration, Sales, Underwriting and Pricing. |
| CPS020 | Secure coding practices | Please make sure to validate all input from untrusted sources (e.g. user input or external systems) at least for reasonable max length and for malicious content where applicable. <br/><br/> Make sure sensitive information is not included in responses, exceptions and logs. Large values should also not be included in those places as this allows attackers to flood the system or increase our infrastructure costs. <br/><br/> Any security related data (e.g. passwords, tokens) that is being stored permanently must be encrypted. <br/><br/> User file uploads should be scanned with antivirus software. <br/><br/> Json deserialization should not be using TypeNameHandling = TypeNameHandling.Auto. If that is required a custom Serialization Binder must be provided. <br/><br/> Another common issue is ReDoS. Please read the article for more details [Preventing Regular Expression Denial of Service (ReDoS)](https://www.regular-expressions.info/redos.html)  but the short summary is to always make RegEx Compiled, with reasonable Timeout and validate any user input we will be processing with it. <br/><br/> Multi-tenant authorization must also be explicitly verified for every endpoint that accepts tenant identifiers (route, query, headers, or body): either enforce support-user/system-user access, or require authenticated user tenant to match requested tenant and return HTTP 403 on mismatch. Do not silently auto-correct tenant identifiers for authorization decisions. <br/><br/> Reviewer quick-check for tenant-boundary endpoints: verify a mismatch check exists for non-support users, verify action-level tenant source is trusted before any SetSystemUserContext call, and verify any SetSystemUserContext use is only for legitimate system operations. |
| CPS021 | Data Classification | Please add any new columns to the list [Data classification](https://EShopinsure.atlassian.net/wiki/x/yLeC). <br/><br/> Please also review if the list of tables and columns (and any other persistent data like S3) might be missing any previously added/changed items. <br/><br/> Keys (both primary and foreign), dates like “created on”, flags like “enable emails” and standard columns like “TenantId/Scope” can be skipped. |
| CPS022 | Live data (data patching/configuration/permissions) | Every code fix should be considered if previous instances of the root cause (or old version of the code in general) could have created bad data in Staging or Production. There might be additional [data patching process](https://EShopinsure.atlassian.net/wiki/x/pwBBBg), re-publishing of failing integration messages or other manual activities that need to be identified and coordinated with the release. E.g. when stopping null fields could there be entries already created with nulls. If there were messages failing, should we re-publish them from error queue or can we trigger some action that will publish similar message fixing up the data previously created. <br/><br/> If there are any new permission checks added to an existing API endpoint, make sure that tenants using the endpoint are informed so that they may grant the new permissions to relevant user roles if needed. <br/><br/> Another common case is additional configuration required after the changes are deployed, e.g. set Tenant time zone after new field is added. <br/><br/> Please consider if any of such activities could be automated with a migration-like code or some logic on application startup. When there is a manual activity needed please create a task and assign to the support team and add [UPDATE DATA] tag as a reminder of changes needed on Staging and Production. |
| CPS023 | No confidential information | No passwords, API keys, real credentials, tenant identifiable information (real tenant names, addresses, emails, phone numbers, etc.), or personally identifiable information (real person names, addresses, emails, etc.) in source code, tests, documents, logs, etc. These must be substituted with fake data, e.g. Tenant A, Company B, Admin user, etc. |
| CPS024 | Resilient communications | All HTTP communication between services needs to follow common resilience practices. The usual set includes retry, timeout handling, circuit breaker, throttling (429). Always treat external services with limited trust and prepare the system for graceful handling of cases when they do not work as excepted. <br/><br/> Take extra care of integration event consumers to allow for out-of-order message delivery, retries and safe handling of idempotence (processing the same message twice should not cause errors). |
| CPS025 | High-value automation opportunities | Please consider whether any activities created by the feature or behavior changed in this pull request are repeatable and would bring significant value if captured in a script, agent skill or agent instructions. This checklist item is not about generic pull request administration and not about everything that could theoretically be automated; it applies only where automation would materially reduce the chance of serious errors, save a lot of time, or both. <br/><br/> Good candidates are repetitive manual steps that are easy to get wrong, time-consuming investigation or verification flows, or release/support activities that are likely to recur when the same type of feature or change is delivered again. |
| CPS026 | Self-review | You have reviewed important areas of the code changes before publishing the pull request. It’s recommended to do self-review using one or more AI-assisted tools. Please address this checklist item with a summary of important code areas that have been self-reviewed. <br/><br/> The following are some aspects that are often checked for in self-review <br/><br/> - Readability and maintainability. Also check for extendibility if the feature is expected to be extended in the future <br/> - Security <br/> - Performance: CPU and memory consumption <br/> - BDD feature files: typos, grammar errors, missing scenarios, merging similar scenarios into a scenario outline, etc. |
